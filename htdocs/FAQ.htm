FAQ  
 

Introduction

What is dTV ?

dTV is a piece of software that grabs analog, interlaced video, and deinterlace it to make it a progressive scan feed for use on your computer monitor or projector.

Analog, interlaced video is what output devices like vcr, laserdisc, game consoles ...


What is so special about dTV ?

Unlike most existing software that tries to do the same, dTV implements very sophisticated algorithms to deinterlace such sources. Using such algorithms makes sure that each kind of video source is handled a different way. It provides the user with a level of image quality unknown to PC owners up to now.

Some of those techniques are ;

  • video source deinterlacing
  • inverse telecine detection
  • 2:2 (for pal movies) as well as 3:2 (for ntsc movies) pulldown

The image quality resulting from those complex operations is high and allows some people equipped with data capable projectors to feed them with a progressive scan source that was interlaced right before.


Who made what ?

See dTV -- Authors for the main contributors and dTV -- Thanks for a list of people who have helped make dTV a success.


General issues

Should I use dTV to deinterlace the output from my set-top player ?

While it's feasible and works quite fine (see the "dTV WinDVD challenge" in the screenshots section), home theater PC's have shown the best results while using dvd decoding software along with a dvd-rom and a modern graphic card.
However, it's been used quite often for test purposes (testing of grab quality on a high quality source, testing pulldown techniques on well encoded sources etc..).

Can I remote control dTV ? YES

Yes you can. dTV is being developed mainly by people using it as an analog source scaler for their projector and they prefer for sure using dTV from their remote on the sofa rather than from their desk ;-)

dTV is thus filled with keyboard shortcuts for all main options so that you can do everything quickly.
More, we suggest the use of Girder from Ron Bessems as the application of choice for remote controlling your computer.

Hardware issues

Can I use any vga card as a display device ?

Despite this statement from Mark Rejhon ;

"dTV requires an AGP card. It is impossible to do on a PCI video card. dTV uses massive amounts of *bi-directional* bus bandwidth: 250 megabits per second incoming bandwidth (from TV card) and about 500 megabits per second outgoing bandwidth (to video card). This bus data firehose is not feasible on PCI bus, as PCI can reliably continuously transfer up to about 600-700 megabits per second *in one direction only* out of a possible 1056 megabits per second (132 megabytes per second) due to overhead. Try to go bi-directional, and framerate literally gets the death penalty!"

However, experience in the last months showed the some users were able to use their PCI card with success and without dropped frames. In some cases however, frame rate is going down when higher and higher resolutions are tried ;

IanD reported I have an 8MB 3dfusion PCI on a BX mobo Celeron 566 64MB RAM 66MHz FSB, with Dynalink Magic TView (Conexant 878) and can only execute dTV at 16 bit colour depth. When running a 1024x768 desktop, dTV v1.9 with a 720x576 @ 50Hz PAL composite input gives 0 DFS in manual Video De-interlace mode for dTV window sizes up to 848x629 (as determined from dtv.ini) but 7+ DFS for window sizes even 1 pixel above either of these dimensions

Bottom line ; no specific requirement for dTV. Probably works not so bad on any vga card. However, as of today, GeForce based cards or (dualhead) Matrox G400 are successfully used by a lot of users.


Which capture card can I use ?

Almost any card (and that means A LOT) based of former Brooktree (now Conexant) chips including bt848/878/879. Why those ? They are, for several years now, the chipsets that are easy to program (there has been A LOT of software based on those) and very cheap. Side effect ; a great number of manufacturers have (and are still) produced cards based on those chipsets.

While we can't claim "this card is the one you need", let's give you a couple ones that, at least, have been successfully used by some users ;

  • Zoltrix Genie TV http://www.zoltix.com/products/GenieTVspecs.htm
  • The WinTV series from Hauppage http://www.hauppage.com/html/products.htm
  • Guillemot maxi video 3 http://fr.guillemot.com/brand/gamme/video/mtvvidp3/mtvvidpci3.html
  • The Pinnacle Studio PCTV http://www.pinnaclesys.com/ProductPage.asp?Product_ID=6

Take a look at the card list. This will never be complete so feel free to drop us an email reporting your experience with dTV and your card.

This list is originally based on the linux bttv driver on which the program was originally based.

 

Can I use my existing video card's video input ?

Probably not. dTV is supporting (see previous question) the Brooktree/Conexant chipset. Supporting others chipsets, while feasible, means some hard work and some solid knowledge of those chipsets. And unfortunately, no recent vga card with video-in feature includes Brooktree/Conexant chipset. Gx00, Asus Deluxe etc... are thus not supported at the moment.

If you have some programming knowledge regarding the grab chipsets used in those card, you might want to help us. See the "How can I help" section.


Can I use more than one video input ?

Some "nut" people (he's admitting it himself ;-) who are playing with Picture In Picture a lot have asked if dTV could drive multiple cards or multiple inputs. A card allowing at worst 4 video inputs is manufactured by Leadtek. It's the Winview 605.

It is Brooktree based so it might work with dTV but clearly nothing is done right now to drive multiple video inputs at the same time.

See the associated topic on AVS forum.

Troubleshooting

What is this "purple stripes" issue ?

UPDATE: Michael Eskin of Conexant Systems Inc. has fixed this issue with an adjustable INI file setting, as of dTV version 2.0.1

When feeding some analog sources to the combination grab card/dTV, some numerous horizontal colored lines appear on the image.

See this picture where macrovision has been added to the video signal. The same video signal without the macrovision protection is perfectly stripes-free.

This phenomenon seems to be, at least, more visible when a macrovision protected video signal is fed to some grab cards but this has been reported also (despite less visible) on unprotected material.

Other examples of this phenomenon can be seen here ;
avs:MacroVision side effect
avs:dTV Purple Stripes - is this capture card dependent?

Some info from the Dvd Faq ;

"Macrovision adds a rapidly modulated colorburst signal ("Colorstripe") along with pulses in the vertical blanking signal ("AGC") to the composite video and s-video outputs. This confuses the synchronization and automatic-recording-level circuitry in 95% of consumer VCRs. Unfortunately, it can degrade the picture, especially with old or nonstandard equipment. Macrovision may show up as stripes of color, distortion, rolling, black & white picture, and dark/light cycling. Macrovision creates problems for many line doublers."

WARNING This has been reported only when feeding, for test purposes, a dvd player to the computer. Problems on other type of sources have yet to be shown.

So in common cases, you don't have to worry about this.

I have slower system and  I get some dropped frames ?

Some people experienced some dropped frames or extreme jerkiness on slower machines, say less than 500 mhz. Some performance tuning parameters for this are available in the [Threads] section of the .ini documentation. Depending upon your graphics card you may also try running with a lower screen resolution or color depth. Also try to limit what else is running at the time, including System Tray applications that may cause frequent interupts.

Despite a very powerful system, I've still got some dropped frames ?

Some people experienced some dropped frames (signs of a machine that has problems keeping up with the workload of dTV) even on powerful systems. An hypothesis is that a motherboard agp speed issue might be the cause of such behavior. 

JackB reported We looked at his BIOS settings and the only major difference is that his had a 4X AGP mode and my MB only has a 2X AGP mode We put my video card, a GeForce2 MX, in his (slower supposedly) system and it ran 0 DFS. So it's not the MX throughput that's the problem. So, I'm down to it's my motherboard not being up to speed in either AGP acceleration or I have some settings set incorrectly


I get "overlay update" errors

This will occur if you have an MPACT2 based card running in 24bit mode.  To use dTV with this card you must use a 16bit desktop mode.

Sometimes you get this error message Error calling overlay update. D:/source/deinterlace/dtv/other.c at line ???
We don't have any full explanation of the problem but it happens when resizing dTV or making it go from fullscreen to window mode. Also the color depth, frequency and size of the desktop might be related.
The problem seems to have disappeared with dTV starting from version 2.0.0

Some more info on this issue in this avs topic

How can I help ?

Testing

More and more people are giving dTV a try. Why don't you ?
We'd like to know about your experience with it. What does (not) work ? What can be improved, what feature you'd like to see. All feedback is useful.

Please report bugs on the AV Science Forum after checking (using the search feature) that the problem has not already been reported. Each bug report should include the hardware you're using, especially video card, grab card and processor type and speed.

Coding

Any help is welcome. Despite including very complex code, there's a lot of features in it that are simple or that could be easily implemented without knowing anything about 3:2 pulldown detection.

dTV could especially use some more programmers interested in adding advanced sound or closed captioning features.  

If you've got an idea, or feel like to trying to implement something that is already in the Todo list, contact the authors and join the mailing list. All relevant info to get you started is on the developers page.
